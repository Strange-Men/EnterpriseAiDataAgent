# v1.0.0 Architecture Optimization Plan

> Date: 2026-06-03  
> Scope: Enterprise AI Data Agent v1.0.0 architecture foundation  
> Status: Active

## 项目定位

Enterprise AI Data Agent 是面向 AI Data Analyst / Data Engineer 场景的企业级数据分析平台。当前核心闭环是：

1. 上传 CSV / Excel 数据到 DuckDB。
2. 在 SQL Workspace 中查询、解释、导出和管理数据。
3. 通过 AI Analyst 生成 SQL、解释结果、生成洞察、推荐图表、检测异常和执行多步分析。
4. 用 trace、guardrails、token budget、evaluation harness 支撑可审计的 AI 分析流程。

v1.0.0 不应把未来 LangChain Agent、RAG、知识图谱能力包装成已完成能力；当前目标是把 SQL Workspace + AI Analyst 打磨成稳定、可展示、可扩展的产品骨架。

## 主要问题

### P0: 状态源不一致

- `CURRENT_SESSION.md` 停在 v0.9.8，`backend/VERSION` 是 v0.9.9，`frontend-react/package.json` 是 0.9.4，`package-lock.json` 是 0.3.10。
- README 路线图仍把 v0.9.x 标记为 Current。
- 根布局 metadata 曾写成 Multi-Agent，但当前项目规则禁止主动落多 Agent/RAG。

影响：项目审阅时会显得版本治理不可信，也容易让后续发布和测试报告失焦。

### P1: 前端服务端状态和本地状态混用

- React Query 已安装并挂载 Provider，但系统状态、表列表、AI 状态仍主要由手写 `useEffect`、轮询和 Zustand 管理。
- 多个面板重复触发表加载和状态同步。
- Zustand 适合工作区 UI 状态，但不适合作为所有服务端数据缓存。

优化方向：React Query 管服务端状态，Zustand 管布局、当前 SQL、面板状态、持久化用户偏好。

### P1: 大文件控制器过重

- `frontend-react/src/panels/ai-analysis-panel.tsx` 超过 40KB。
- `frontend-react/src/panels/sql-workspace-panel.tsx` 超过 30KB。
- `frontend-react/src/services/api.ts` 超过 40KB。
- `backend/services/ai_analyst.py` 和 `backend/services/ai_pipeline.py` 承担 LLM 调用、解析、业务编排、异常处理等多重职责。

优化方向：按 feature slice 拆分，而不是按“工具函数桶”继续堆叠。

### P1: AI 分析“没效果”的根因

- LLM 输出解析已有修复，但解析、降级、质量反馈没有形成统一 error contract。
- 前端流式结果、结构化结果、fallback 文本之间的呈现逻辑分散。
- AI 成功与“有业务价值”不是同一件事，当前 UI 对低置信度、空洞输出、数据不足的提示不够明确。

优化方向：统一 `AiResultEnvelope`，把 `status/error/quality_gates/trace/partial_data` 作为稳定契约；前端用统一组件展示完整、部分成功、失败和低置信度状态。

### P2: 文件系统治理偏差

- 根目录存在 `.env`、`.coverage`、`.pytest_cache/`、`.idea/` 等本地文件或目录。
- `backend/data/enterprise.duckdb` 与文件系统规则冲突，规范位置应是根部 `data/`。
- 源码目录中存在 `__pycache__/`。

优化方向：不直接删除用户数据文件；先确保 `.gitignore` 覆盖，再提供非破坏性清理与 DB 迁移步骤。

### P2: 后端数据访问层边界不足

- 路由层做了表名验证，但底层 `DatabaseManager` 仍直接拼 SQL 标识符。
- `DatabaseManager` 默认路径和 `backend.config.DUCKDB_PATH` 各自读取环境变量，配置入口不够单一。
- 查询执行、分页、导出、历史记录和 JSON 序列化契约需要进一步集中。

优化方向：新增 data access facade，统一表名校验、DuckDB 连接、只读执行、分页响应和错误归一化。

## v1.0.0 已落地基础

- 后端版本、前端 package、package-lock、README、AGENTS、CLAUDE 同步到 v1.0.0。
- 新增 `backend/utils/llm_json.py`，集中 LLM JSON 解析。
- 新增 `tests/test_llm_json.py`，覆盖 plain JSON、markdown-wrapped JSON、embedded JSON 和失败路径。
- 新增 `frontend-react/src/hooks/use-server-state.ts`，把系统状态、表列表、AI 状态迁入 React Query。
- `use-system-status.ts` 和 `use-tables.ts` 改为 React Query 驱动，同时兼容现有 Zustand store。
- 根布局 metadata 改为当前真实定位。

## 高星项目借鉴

本轮参考的是同类开源数据产品的架构方向，而不是照搬它们的技术栈：

| Project | 参考点 | 对本项目的落地 |
|---------|--------|----------------|
| [Apache Superset](https://github.com/apache/superset) | SQL Editor、semantic layer、cache、API customization、cloud-native scale | 保留 SQL Workspace 核心体验；增强语义层、查询缓存、API 分层和扩展边界 |
| [Metabase](https://github.com/metabase/metabase) | 易用 BI、embedded analytics、国际化、权限/安全治理 | 前端降低复杂度，AI 结果要以业务用户可理解的方式呈现；继续强化 i18n 和安全边界 |
| [Evidence](https://github.com/evidence-dev/evidence) | SQL + markdown 的 code-based BI，查询结果驱动组件渲染 | 后续报告和分析结果应走 typed data product 思路，而不是只堆 Markdown 文本 |
| [Apache ECharts](https://github.com/apache/echarts) | 大型可视化库的插件化、扩展包和强交互图表能力 | 现阶段保留 Recharts；后续可把 chart renderer 抽象成插件接口，必要时引入 ECharts adapter |

结论：v1.0.0 不需要换掉 Next.js + FastAPI + DuckDB。更关键的是把项目改成“可扩展的数据产品骨架”：server-state cache、feature API modules、AI result envelope、semantic/query/cache 层、图表 renderer 插件边界。

## 推荐技术栈基线

保留：

- Frontend: Next.js 15 + React 19 + TypeScript
- Backend: FastAPI + Uvicorn
- Analytical DB: DuckDB
- AI Provider: Anthropic SDK
- UI: TailwindCSS + lucide-react + Recharts

强化：

- React Query: 服务端状态、缓存、重试、失效、后台刷新
- Zustand: 只保留本地 UI / workspace / persisted preference
- Pydantic v2: API request / response contract
- pytest + Vitest + Playwright: 分层验证
- OpenAPI contract: 前后端类型生成可作为 v1.0.x 后续任务

暂不引入：

- LangGraph / Multi-Agent / RAG / 知识图谱：需要产品需求、数据权限、文档摄取、评测基准明确后再进入 v1.1+。
- 新后端框架：FastAPI 已满足当前平台需求，问题不在框架本身。

## 后续执行路线

### Phase 1: Skeleton Stabilization

- 拆分 `frontend-react/src/services/api.ts` 为 `api/http-client.ts`、`api/tables.ts`、`api/query.ts`、`api/ai.ts`、`api/streams.ts`。
- 已完成第一步：`api/http-client.ts`、`api/status.ts`、`api/tables.ts`，旧 `services/api.ts` 保持兼容 re-export。
- 拆分 `ai-analysis-panel.tsx` 为 mode handlers、view components、state hook。
- 拆分 `sql-workspace-panel.tsx` 为 editor shell、query runner、AI SQL generator、result toolbar。
- 增加 API response envelope 类型。

### Phase 2: AI Effectiveness

- 为 AI SQL、insights、semantics、anomalies 建立统一质量评分和 fallback 呈现。
- 把 trace 和 quality gates 在 UI 中作为一等信息展示。
- 增加真实数据集 golden evaluation 报告，不只测 mock endpoint。

### Phase 3: Performance

- Query result 默认分页消费，避免大结果一次进入 React state。
- AI streaming 内容按 requestAnimationFrame 或节流批量更新。
- Monaco 预加载策略和 schema cache 进一步下沉到 dedicated hook。
- 用 Playwright 做大表 FPS、DOM 节点、内存快照基准。

### Phase 4: Data Governance

- 提供 DB path migration script，将误放在 `backend/data/` 的 DuckDB 文件迁移到 `data/`。
- 清理本地缓存和生成物，但不删除用户数据。
- 增加 pre-commit style repository health check。

## 验收标准

- `python -c "from backend.main import app"` 通过。
- `pytest` 通过。
- `cd frontend-react && npm run type-check` 通过。
- `cd frontend-react && npx next build` 通过。
- Git status 中不包含 `.env`、`node_modules`、`.duckdb`、`__pycache__` 等应忽略文件。

# v0.5.4 — AI System Engineering: Stability, Observability, Guardrails

**Phase transition**: v0.5.3 delivered AI capabilities. v0.5.4 makes them **production-grade**.

No new AI features. No new UI. Focus: **stability, observability, controllability**.

## 1. Current State Audit

### 1.1 Prompt Problems (the "Prompt Soup" risk)

Current state: 8 prompts scattered across files.

| Prompt | Location | Line | max_tokens |
|--------|----------|------|------------|
| `SQL_GENERATION_SYSTEM` | `ai_analyst.py:95` | 95 | 512 |
| `EXPLANATION_SYSTEM` | `ai_analyst.py:107` | 107 | 1024 |
| `INSIGHTS_SYSTEM` | `ai_analyst.py:122` | 122 | 1024 |
| `CHART_SUGGESTION_SYSTEM` | `ai_analyst.py:148` | 148 | 512 |
| `SEMANTICS_SYSTEM` | `ai_analyst.py:392` | 392 | 1024 |
| `SUGGESTED_QUESTIONS_SYSTEM` | `ai_analyst.py:466` | 466 | 512 |
| `ANALYSIS_PLAN_SYSTEM` | `ai_analyst.py:545` | 545 | 1024 |
| `_SUMMARY_SYSTEM` | `ai_pipeline.py:94` | 94 | 512 |

**Problems**:
- Prompts live in service files, not separated
- No version tracking or A/B testing capability
- Context injection is ad-hoc (string concatenation scattered in each function)
- No formal "prompt contract" — what variables does each prompt expect?
- `_SUMMARY_SYSTEM` is in the pipeline file, not the analyst file — inconsistency

### 1.2 Token Budget Problems

Current state: `max_tokens` is set per call, but **input tokens are unbounded**.

| Function | Input risk |
|----------|-----------|
| `build_schema_context` | No column limit — 100+ columns = 2000+ input tokens |
| `_build_explain_user_msg` | 50 result rows JSON — could be 10K+ tokens |
| `generate_insights` | 50 result rows JSON — same risk |
| `generate_semantics` | Columns truncated to 20 ✓, rows to 5 ✓ |
| `generate_analysis_plan` | Columns truncated to 20 ✓, rows to 5 ✓ |
| `run_autonomous_analysis` | Calls `generate_sql` up to 6 times — no total budget |
| `build_follow_up_context` | 5 rows + schema — budgeted to ~220-500 tokens ✓ |

**Problem**: No total token budget for multi-step workflows. A 6-step autonomous analysis could consume 6 × (schema + data) with no upper bound.

### 1.3 Guardrail Problems

Current state: **No guardrails at all**.

- No max steps enforcement (plan says max 6, but code just `[:6]` truncates)
- No max SQL queries per analysis
- No timeout on autonomous analysis
- No recursion protection (step A depends on B depends on A = infinite)
- No failed-step threshold (all 6 steps could fail, system still generates summary)
- No circuit breaker on repeated LLM errors

### 1.4 Observability Problems

Current state: **Zero observability**.

- No logging of LLM calls
- No token counting
- No timing breakdown per step
- No error categorization
- No way to debug "why did the AI generate this SQL?"
- No trace record for autonomous analysis

### 1.5 Testing Problems

Current state: **No AI evaluation**.

- No golden question set
- No SQL quality measurement
- No hallucination detection
- No regression testing when prompts change
- No way to know if a prompt change improves or degrades quality

## 3. Module 2 — Token Budget System

### 3.1 Goal

Every LLM call has a **measured, enforced** input token budget. No unbounded inputs.

### 3.2 Token Estimation

We don't need a tokenizer — a simple heuristic is sufficient:

```python
# backend/prompts/token_budget.py

def estimate_tokens(text: str) -> int:
    """Rough token estimate: ~4 chars per token for English, ~2 for CJK-heavy."""
    # Conservative: 3.5 chars per token
    return len(text) // 3
```

### 3.5 Workflow Token Cap (Multi-step)

For `run_autonomous_analysis`, add a `WorkflowTokenTracker`:

```python
class WorkflowTokenTracker:
    def __init__(self, total_budget: int = 25000):
        self.total_budget = total_budget
        self.consumed = 0
        self.calls: list[dict] = []

    def can_proceed(self, estimated_input: int, estimated_output: int) -> bool:
        return (self.consumed + estimated_input + estimated_output) <= self.total_budget

    def record(self, input_tokens: int, output_tokens: int, operation: str):
        self.consumed += input_tokens + output_tokens
        self.calls.append({"operation": operation, "in": input_tokens, "out": output_tokens})
```

If the budget is exhausted mid-workflow, the remaining steps are marked `"skipped"` with reason `"token_budget_exceeded"`.

## 4. Module 3 — Runtime Guardrails

### 4.1 Goal

Autonomous analysis cannot run unbounded. Every workflow has configurable limits.

### 4.2 Guardrail Configuration

```python
# backend/services/guardrails.py

@dataclass
class AnalysisGuardrails:
    max_steps: int = 6                    # max plan steps
    max_sql_queries: int = 8              # max SQL executions per analysis (includes retries)
    max_consecutive_failures: int = 2     # stop after N consecutive step failures
    max_total_time_seconds: int = 120     # hard timeout for entire analysis
    max_step_time_seconds: int = 30       # timeout per step
    max_recursion_depth: int = 2          # max depends_on chain length
    require_minimum_success: int = 1      # at least N steps must succeed

DEFAULT_GUARDRAILS = AnalysisGuardrails()

STRICT_GUARDRAILS = AnalysisGuardrails(
    max_steps=4,
    max_sql_queries=5,
    max_consecutive_failures=1,
    max_total_time_seconds=60,
    max_step_time_seconds=15,
)
```

### 4.3 Guardrail Checks

```python
class GuardrailViolation(Exception):
    def __init__(self, rule: str, message: str, context: dict | None = None):
        self.rule = rule
        self.message = message
        self.context = context or {}
        super().__init__(f"Guardrail [{rule}]: {message}")

class AnalysisGuard:
    def __init__(self, config: AnalysisGuardrails):
        self.config = config
        self.start_time = time.time()
        self.steps_executed = 0
        self.sql_queries = 0
        self.consecutive_failures = 0

    def check_before_step(self, step_def: dict, executed_steps: list[dict]):
        """Called before executing each step."""
        # 1. Max steps
        if self.steps_executed >= self.config.max_steps:
            raise GuardrailViolation("max_steps", f"Reached {self.config.max_steps} steps")

        # 2. Max SQL queries
        if self.sql_queries >= self.config.max_sql_queries:
            raise GuardrailViolation("max_sql_queries", f"Reached {self.config.max_sql_queries} SQL queries")

        # 3. Total timeout
        elapsed = time.time() - self.start_time
        if elapsed > self.config.max_total_time_seconds:
            raise GuardrailViolation("total_timeout", f"Analysis exceeded {self.config.max_total_time_seconds}s")

        # 4. Recursion depth
        depth = self._chain_depth(step_def, executed_steps)
        if depth > self.config.max_recursion_depth:
            raise GuardrailViolation("recursion_depth", f"Dependency chain depth {depth} exceeds {self.config.max_recursion_depth}")

        # 5. Consecutive failures
        if self.consecutive_failures >= self.config.max_consecutive_failures:
            raise GuardrailViolation("consecutive_failures", f"{self.config.max_consecutive_failures} consecutive failures")

    def record_step_result(self, success: bool):
        self.steps_executed += 1
        self.sql_queries += 1
        if success:
            self.consecutive_failures = 0
        else:
            self.consecutive_failures += 1

    def check_after_all(self, executed_steps: list[dict]):
        """Called after all steps. Enforces minimum success."""
        successful = sum(1 for s in executed_steps if s["status"] == "success")
        if successful < self.config.require_minimum_success:
            raise GuardrailViolation(
                "minimum_success",
                f"Only {successful} steps succeeded (minimum: {self.config.require_minimum_success})"
            )

    def _chain_depth(self, step_def: dict, executed_steps: list[dict]) -> int:
        """Calculate dependency chain depth."""
        depth = 0
        current = step_def
        visited = set()
        while current.get("depends_on") is not None:
            dep = current["depends_on"]
            if dep in visited:
                return 999  # circular dependency
            visited.add(dep)
            depth += 1
            prev = next((s for s in executed_steps if s["step"] == dep), None)
            if prev is None:
                break
            current = prev
        return depth
```

### 4.4 Integration Points

`run_autonomous_analysis` and `run_autonomous_analysis_stream` wrap the step loop with guardrail checks:

```python
guard = AnalysisGuard(guardrails or DEFAULT_GUARDRAILS)

for step_def in plan:
    try:
        guard.check_before_step(step_def, executed_steps)
    except GuardrailViolation as v:
        executed_steps.append({
            "step": step_def["step"],
            "status": "skipped",
            "reason": v.rule,
            "error": v.message,
        })
        continue  # or break, depending on severity

    # ... execute step ...

    guard.record_step_result(success=True/False)

guard.check_after_all(executed_steps)
```

### 4.5 Fallback Strategies

| Failure type | Fallback |
|-------------|----------|
| Step SQL error | Skip step, continue with remaining |
| Consecutive failures (2) | Stop execution, generate summary from successful steps |
| Token budget exceeded | Skip remaining steps, mark as `"skipped"` |
| Total timeout | Stop execution, return partial results |
| Circular dependency | Break chain, execute step without dependency context |
| All steps failed | Return error with plan, no summary |

### 5.3 Trace Recorder

```python
class TraceRecorder:
    """Thread-local trace recorder. One recorder per analysis."""

    def __init__(self, question: str, table: str, mode: str, language: str):
        self.trace = AnalysisTrace(
            trace_id=str(uuid.uuid4())[:8],
            question=question,
            table=table,
            mode=mode,
            language=language,
            start_time=time.time(),
        )

    def record_llm_call(
        self,
        operation: str,
        phase: str,
        prompt_name: str,
        input_text: str,
        output_text: str,
        latency_ms: float,
        status: str,
        error: str | None = None,
        sql: str | None = None,
        step: int | None = None,
    ):
        input_tokens = len(input_text) // 3
        output_tokens = len(output_text) // 3
        event = TraceEvent(
            timestamp=time.time(),
            operation=operation,
            phase=phase,
            prompt_name=prompt_name,
            input_tokens=input_tokens,
            output_tokens=output_tokens,
            latency_ms=latency_ms,
            status=status,
            error=error,
            sql=sql,
            step=step,
        )
        self.trace.events.append(event)
        self.trace.total_input_tokens += input_tokens
        self.trace.total_output_tokens += output_tokens
        self.trace.total_llm_calls += 1

    def record_guardrail_violation(self, rule: str, message: str):
        self.trace.guardrail_violations.append(f"{rule}: {message}")

    def finish(self, status: str = "success"):
        self.trace.end_time = time.time()
        self.trace.status = status
        return self.trace

    def to_dict(self) -> dict:
        """Serialize for JSON response."""
        ...

    def save_to_file(self, directory: str = "logs/traces/"):
        """Persist trace to JSON file for later analysis."""
        ...
```

## 7. Implementation Order

```
Module 1 (Prompts)  ──→  Module 2 (Token Budget)  ──→  Module 3 (Guardrails)
         │                                                          │
         │                                                          ↓
         └──────────────────────────────────→  Module 4 (Trace)
                                                       │
                                                       ↓
                                                Module 5 (Eval)
```

**Rationale**:
1. Module 1 first — everything else references prompts by name, so prompts must be in place
2. Module 2 second — token budgeting needs prompt contracts to know which budget applies
3. Module 3 third — guardrails need token budget to enforce workflow caps
4. Module 4 fourth — trace needs all the above to record accurately
5. Module 5 last — evaluation needs trace to measure quality

### 7.1 Per-Module Steps

**Module 1 (1-2 hours)**:
1. Create `backend/prompts/` with all prompt files
2. Define PromptContract in each file
3. Move prompts from `ai_analyst.py` and `ai_pipeline.py`
4. Create user message builders
5. Update imports in services
6. Verify: `python -c "from backend.main import app"`
7. Verify: `npx tsc --noEmit`

**Module 2 (1-2 hours)**:
1. Create `backend/prompts/token_budget.py`
2. Add `estimate_tokens()`, truncation functions
3. Create `WorkflowTokenTracker`
4. Integrate truncation into `ai_analyst.py` functions
5. Integrate tracker into `ai_pipeline.py`
6. Verify: existing tests pass
7. Test: manually verify token estimates are reasonable

**Module 3 (1 hour)**:
1. Create `backend/services/guardrails.py`
2. Integrate into `run_autonomous_analysis` and stream variant
3. Test: create a scenario that triggers each guardrail
4. Verify: streaming still works with guardrail interruptions

**Module 4 (1-2 hours)**:
1. Create `backend/services/trace.py`
2. Add trace params to `_call_llm`/`_call_llm_stream`
3. Create TraceRecorder in pipeline functions

**Module 5 (1-2 hours)**:

## 8. Token Strategy Summary

| Operation | Input Budget | Output Budget | Key Constraint |
|-----------|-------------|---------------|----------------|
| SQL Generation | 3,000 tokens | 512 tokens | Schema max 2,000 tokens |
| Explanation | 6,000 tokens | 1,024 tokens | Results max 30 rows |
| Insights | 5,000 tokens | 1,024 tokens | Results max 30 rows |
| Chart Suggest | 3,000 tokens | 512 tokens | Results max 10 rows |
| Semantics | 3,000 tokens | 1,024 tokens | Columns max 20 |
| Questions | 2,000 tokens | 512 tokens | No sample rows |
| Planning | 3,000 tokens | 1,024 tokens | Columns max 20, rows max 5 |
| Summarizer | 3,000 tokens | 512 tokens | Steps max 6, 3 samples each |

## 9. Guardrail Strategy Summary

|------|---------|--------|-----------------------|
| max_steps | 6 | 4 | Skip remaining steps |

| recursion_depth | 2 | 1 | Break chain |

## 10. Risk Analysis

| Risk | Impact | Mitigation |
|------|--------|-----------|
| Prompt migration breaks existing behavior | HIGH | Copy prompts exactly, test before removing originals. Each prompt file is independently testable. |
| Token estimation too inaccurate | MEDIUM | Heuristic is conservative (rounds up). Can swap to tiktoken later without API change. |
| Guardrails too aggressive | MEDIUM | Defaults are permissive. Strict mode is opt-in. Can tune per-request. |
| Trace overhead slows analysis | LOW | Trace is lightweight (just string lengths and timestamps). No I/O in hot path by default. |
| Golden questions become stale | MEDIUM | Review quarterly. Number them for tracking. |
| Circular dependency detection misses edge cases | LOW | Depth-based check catches all loops. Max depth 2 means any cycle is broken. |
